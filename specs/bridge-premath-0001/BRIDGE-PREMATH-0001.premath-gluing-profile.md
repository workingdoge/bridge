## Language

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**,
**SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this
specification are to be interpreted as described in RFC 2119 and RFC 8174.

## 1. Purpose

This document indexes the current bridge + secret-management suite as one
Premath-admissible object.

For a reader-facing fibre-bundle view of this profile, see
`secret-bound-execution-bundle.md`. That note explains the base context, fibre,
local charts, overlap objects, global sections, and obstruction fixtures without
replacing the normative profile below.

Its job is not to replace:

- Bridge Spec v0.2,
- bridge executable policy,
- bridge adapter contract,
- SECRET-0001,
- SECRET-0002,
- SECRET-0003,

but to make their common context, pullbacks, overlap objects, and gluing laws
explicit.

A conforming implementation SHALL expose:

1. a category of admissible execution contexts,
2. a coverage generated by chart projections,
3. pullback/reindexing along admissible refinements,
4. a glued global object assembled from compatible local charts, and
5. explicit declarations for any optional pushforward structure.

## 2. Relationship to Premath

This profile is a realization profile for the Premath kernel doctrine.

The Premath kernel requires:

- a category `C` of contexts,
- a coverage `Cov`,
- an indexed assignment `Def : C^op -> V`,
- functorial pullback `f*`,
- locality along covers,
- and contractible descent.

This profile supplies those ingredients for the bridge + secret suite and fixes
their intended interpretation.

### 2.1 Cited premath modules

The premath kernel surface this profile realizes lives in
`fish/sites/premath/` under:

- `specs/raw/asp/ASP0.md` — the Admissible Simplex Presentations
  substrate. Supplies the category of contexts `C` and the
  coverage-building mechanism. Every admissible object in this
  profile's §4 category is an ASP presentation whose role colors are
  the context coordinate families.
- `specs/raw/ident/IDENT0.md` — the first concrete profile over
  `ASP0`. Fixes an ordered master role simplex
  `[issuer, subject, credential, scope, epoch]` with trust,
  binding, grants, freshness, and revocation witnesses. §4.2.1
  Request coordinates (`subject_id`, `witness_jti`, `consumer_id`)
  and §4.2.5 Provider-fact coordinates (`issuer_epoch`) are the
  bridge-side carriers of that simplex.
- `specs/raw/ident/JWT0.md`, `JWT0-OBS.md`, `JWT0-SEARCH.md`, and
  `JWT1-JWKS0.md` — candidate backend realization profiles for
  identity at the bridge + secret seam. A conforming implementation
  MAY adopt these as concrete backend+materializer profiles for the
  identity-class secret-suite slot; doing so is NOT required by this
  profile, but this profile names them as the first concrete place
  identity backends can land.

Citations are path-specific so downstream consumers (bridge
admission profiles, SECRET-0002 instances, workingdoge IAM
realization) can trace any bridge context coordinate back to the
premath module that grounds it.

### 2.2 Identity simplex as first concrete PremathContext example

IDENT0's role simplex `[issuer, subject, credential, scope, epoch]`
is the canonical worked example of how a PremathContext's identity
coordinates decompose.

Under this profile:

- `issuer` is the admission-time authority asserting a credential
  (§4.2.5 `mode_controller_id`, `audit_sink_id`, or the Cloudflare
  Access team handle for workingdoge-class consumers).
- `subject` is the principal the credential binds (§4.2.1
  `subject_id`, `consumer_id`).
- `credential` is the typed credential artifact itself (§4.2.1
  `witness_jti` for JWT backends, or the non-exportable handle for
  key backends).
- `scope` is the admissible operation surface (§4.2.4
  `requested_tool`, `requested_resource`, `requested_method`).
- `epoch` is the freshness / rotation boundary (§4.2.2
  `secret_epoch`, §4.2.5 `issuer_epoch`, `mode_epoch`).

A bridge implementation that consumes the premath identity profile
SHALL expose a forgetful map from its context to the identity
simplex and preserve the IDENT0 witnesses (trust, binding, grants,
freshness, not-revoked) along that map.

## 3. Ambient coherence level

This profile chooses `V = Gpd`.

Semantic sameness `~` SHALL mean isomorphism of normalized admissibility tuples
that preserve the declared overlap identities and invariants.

An implementation MAY additionally expose:

- a stricter `Set` realization after full canonicalization, or
- a richer `Cat` realization once nontrivial proof morphisms inside fibers are
  made first-class.

Core conformance for this document does not require those stronger realizations.

## 4. Category of contexts

### 4.1 Objects

An object `Gamma` of `C` is an admissible secret-bound execution context.

A conforming `Gamma` SHALL contain enough information to determine all of the
following without reading raw plaintext secret values:

- bridge request identity,
- witness identity and bound actor,
- requested tool and requested resource,
- source and destination domain,
- resource-to-secret seam,
- secret object and selected version surface,
- backend and materializer binding,
- requested materialization method and TTL,
- authoritative provider facts,
- deployment profile and provider catalog identity,
- audit stream head or checkpoint identity.

This profile packages that surface in the `PremathContext` interface and schema.

### 4.2 Required context coordinates

A conforming context SHALL contain at least the following coordinates.

#### 4.2.1 Request coordinates

- `request_id`
- `trace`
- `witness_jti`
- `requested_tool`
- `requested_resource`
- `source_domain`
- `destination_domain`
- `subject_id`
- `act_for` when present
- `consumer_kind`
- `consumer_id`

#### 4.2.2 Interpretation coordinates

Bridge interpretation remains bridge-owned, but the realized Premath context
SHALL carry stable references for the admitted meaning assignment:

- `foreign_observation_id`
- `source_context_id`
- `interpretation_binding_id`
- `typed_observation_id`
- `interpretation_status`

#### 4.2.3 Secret coordinates

- `secret_id`
- `secret_class`
- `version_selector` or `version_id`
- `secret_epoch` when available

#### 4.2.4 Resource-to-secret seam

The current suite already contains the ingredients of this seam, but the seam is
not explicit everywhere. This profile therefore REQUIRES a normalized seam
object containing at least:

- `binding_id`
- `requested_resource`
- `secret_id`
- `backend_profile_id`
- `materializer_profile_id`
- `issuance_mode`

#### 4.2.5 Policy coordinates

- `requested_method`
- `requested_ttl_s`
- `mode`
- `conf_level`
- `integ_level`
- `non_exportable`
- `direct_model_access_prohibited`

#### 4.2.6 Provider-fact coordinates

- `deployment_profile_id`
- `provider_catalog_id`
- `authoritative_now`
- `attestation_id`
- `posture_digest`
- `revocation_snapshot_id`
- `issuer_epoch`
- `mode_controller_id`
- `mode_epoch`
- `audit_sink_id`

#### 4.2.7 Audit coordinates

- `stream_id`
- `last_sequence`
- `last_event_hash`

### 4.3 Morphisms

A morphism `f : Gamma' -> Gamma` is an admissible refinement.

Informally, `Gamma'` is narrower or fresher than `Gamma`.

A conforming morphism SHALL be representable as a finite list of refinement
operations such as:

- tightening request scope,
- reducing TTL,
- tightening confidentiality or integrity requirements,
- selecting a more specific secret version,
- restricting backend or materializer binding,
- refreshing provider facts,
- advancing to a more restrictive mode,
- projecting to a chart,
- projecting to an audit stream prefix.

A morphism SHALL NOT:

- widen effective authority,
- increase allowed plaintext surface,
- weaken `non_exportable`,
- re-enable direct model access,
- move from `burn` to a less restrictive mode,
- resurrect a revoked or destroyed secret version,
- or substitute weaker or staler authoritative facts as though they were a
  refinement.

### 4.4 Burn and restore

Entering `safe` or `burn` mode SHALL be modeled as refinement.

Restore SHALL NOT be treated as an ordinary pullback from a burned context.

Restore SHALL instead construct a fresh base context carrying at least:

- a fresh mode epoch,
- a fresh issuer epoch,
- fresh attestation or equivalent posture basis,
- restore authorization basis,
- and a fresh audit continuation point.

This preserves the bridge suite's burn semantics and prevents an implementation
from modeling restore as a silent inverse of burn.

## 5. Coverage

Let `Cov` be the coverage generated by the following chart projections:

- `u_B : Gamma_B -> Gamma` for the bridge chart,
- `u_S : Gamma_S -> Gamma` for the secret-core chart,
- `u_M : Gamma_M -> Gamma` for the materialization chart,
- `u_P : Gamma_P -> Gamma` for the provider-plane chart,
- `u_A : Gamma_A -> Gamma` for the audit chart.

Implementations MUST make pullbacks of these projections available so that
overlaps such as the following can be formed:

- `Gamma_B ×_Gamma Gamma_P`
- `Gamma_B ×_Gamma Gamma_S`
- `Gamma_S ×_Gamma Gamma_M`
- `Gamma_M ×_Gamma Gamma_P`
- `Gamma_B ×_Gamma Gamma_A`
- `Gamma_P ×_Gamma Gamma_A`

The coverage MAY be refined further, but descent MUST remain stable under such
refinement.

## 6. Indexed charts

This profile uses five indexed charts. Each is a functor `C^op -> Gpd`.

### 6.1 `BridgeChart`

`BridgeChart(Gamma)` contains the normalized bridge-side admissibility surface,
including:

- witness,
- authorize request,
- assembled policy input,
- decision,
- and effective authority.

`BridgeChart` SHALL treat effective authority as a meet, never a union.

### 6.2 `SecretCoreChart`

`SecretCoreChart(Gamma)` contains the logical secret and lifecycle surface,
including:

- secret object identity,
- selected version or selector,
- lifecycle state witness,
- materialization request surface,
- and materialization grant surface.

### 6.3 `MaterializationChart`

`MaterializationChart(Gamma)` contains the last-mile realization surface,
including:

- backend profile identity,
- materializer profile identity,
- backend binding,
- plan request surface,
- and materialization session.

### 6.4 `ProviderPlaneChart`

`ProviderPlaneChart(Gamma)` contains normalized authoritative facts, including:

- provider catalog identity,
- deployment profile identity,
- attestation result identity,
- revocation snapshot identity,
- mode state identity,
- provider decision identity,
- and audit readiness basis.

Raw provider payloads are not admissible objects in this chart.
Only normalized provider results and normalized decision surfaces are.

### 6.5 `AuditChart`

`AuditChart(Gamma)` contains the durable evidence surface, including:

- audit envelope identity,
- stream identity,
- sequence,
- previous hash,
- event hash,
- and checkpoint identity when present.

## 7. Overlap objects

The glued object does not identify charts by filename. It identifies them by
shared overlap objects.

### 7.1 `K_req`

`K_req(Gamma)` is the request overlap.

It SHALL contain at least:

- `trace`
- `request_id`
- `witness_jti`
- `requested_tool`
- `requested_resource`
- `source_domain`
- `destination_domain`
- `subject_id`
- `act_for`
- `consumer_kind`
- `consumer_id`

### 7.2 `K_secret`

`K_secret(Gamma)` is the secret/seam overlap.

It SHALL contain at least:

- `secret_id`
- `secret_class`
- `version_selector` or `version_id`
- `secret_epoch` when available
- `binding_id`
- `requested_resource`
- `backend_profile_id`
- `materializer_profile_id`

This makes the bridge-to-secret seam explicit.

In this repo's normalized surface, that seam is concretized by
`SECRET-0002`'s `MaterializationPlanRequest` and `MaterializationSession`
objects.

### 7.3 `K_policy`

`K_policy(Gamma)` is the policy overlap.

It SHALL contain at least:

- `requested_method`
- `requested_ttl_s`
- `mode`
- `conf_level`
- `integ_level`
- `non_exportable`
- `direct_model_access_prohibited`

### 7.4 `K_facts`

`K_facts(Gamma)` is the authoritative-facts overlap.

It SHALL contain at least:

- `deployment_profile_id`
- `provider_catalog_id`
- `attestation_id`
- `posture_digest`
- `revocation_snapshot_id`
- `issuer_epoch`
- `mode_epoch`
- `audit_sink_id`

### 7.5 `K_audit`

`K_audit(Gamma)` is the audit overlap.

It SHALL contain at least:

- `trace`
- `stream_id`
- `sequence` or `last_sequence`
- `payload_hash` or equivalent evidence hash basis
- `prev_event_hash` or `last_event_hash`

## 8. The global definable

For each context `Gamma`, define `Def(Gamma)` to be the groupoid of tuples:

- `(B, S, M, P, A)`

where:

- `B in BridgeChart(Gamma)`
- `S in SecretCoreChart(Gamma)`
- `M in MaterializationChart(Gamma)`
- `P in ProviderPlaneChart(Gamma)`
- `A in AuditChart(Gamma)`

subject to compatibility.

Equivalently, `Def(Gamma)` is the finite-limit object cut out by the overlaps
and compatibility laws listed below.

One convenient presentation is:

```text
Def(Gamma)
  := BridgeChart(Gamma)
   ×_{K_req(Gamma) × K_facts(Gamma)} ProviderPlaneChart(Gamma)
   ×_{K_secret(Gamma) × K_policy(Gamma)} SecretCoreChart(Gamma)
   ×_{K_secret(Gamma) × K_policy(Gamma) × K_facts(Gamma)} MaterializationChart(Gamma)
   ×_{K_audit(Gamma)} AuditChart(Gamma)
```

Implementations MAY realize this by explicit products, dependent records,
indexed stores, or proof-carrying constructors, provided the semantic laws hold.

## 9. Compatibility laws

A tuple `(B, S, M, P, A)` is compatible only if all of the following hold.

### 9.1 Request coherence

The request overlap extracted from the bridge chart SHALL agree with the request
overlap extracted from the provider-plane chart and with the trace carried by the
audit chart.

### 9.2 Seam coherence

The secret/seam overlap SHALL agree across secret-core and materialization, and
it SHALL agree with any bridge-side resource binding used to authorize or plan
the action.

### 9.3 Policy monotonicity

The effective policy surface SHALL be monotone under refinement.

At minimum:

- session TTL SHALL be less than or equal to grant TTL when both are present,
- grant TTL SHALL be less than or equal to bridge effective TTL when both are
  present,
- `non_exportable = true` SHALL be preserved under refinement,
- `direct_model_access_prohibited = true` SHALL be preserved under refinement,
- mode SHALL be equal or more restrictive under refinement,
- and output or egress restrictions SHALL never be widened by a pullback.

### 9.4 Authority-as-meet

The bridge authority surface SHALL be computed as a meet of the relevant local
authorities and constraints.

A compatible tuple SHALL never witness a widening of bridge authority by
combining incomparable allowances across charts.

### 9.5 Provider-fact coherence

The provider-facts overlap SHALL agree across bridge, materialization, and
provider-plane charts after normalization.

Freshness-bounded facts SHALL not be replaced by weaker or stale facts under a
claimed refinement.

### 9.6 Lifecycle coherence

A materialization or session SHALL be compatible only with an admissible secret
version and lifecycle state.

At minimum:

- inactive versions SHALL not materialize,
- revoked or destroyed versions SHALL not materialize,
- and destroyed SHALL remain terminal.

### 9.7 Burn dominance

If any compatible chart places the context in `burn`, the glued object SHALL
carry burn semantics.

In particular:

- no new materialization or allow path may survive as an ordinary allow,
- active sessions SHALL be marked for teardown according to selected policy,
- and the global glued object SHALL remain audit-capable.

### 9.8 Audit-before-allow

A tuple with an allow or session-issue outcome SHALL be compatible only if the
audit surface is durable or the deployment explicitly admits an emergency sealed
queue with equivalent local durability semantics.

### 9.9 No plaintext global section

Raw plaintext secret values SHALL NOT be members of `Def(Gamma)`.

A conforming implementation SHALL treat plaintext or leaf handles as local-only
surfaces outside the global glued object.

## 10. Pullback laws

For every morphism `f : Gamma' -> Gamma`, each chart SHALL expose pullback:

- `f*_B : BridgeChart(Gamma) -> BridgeChart(Gamma')`
- `f*_S : SecretCoreChart(Gamma) -> SecretCoreChart(Gamma')`
- `f*_M : MaterializationChart(Gamma) -> MaterializationChart(Gamma')`
- `f*_P : ProviderPlaneChart(Gamma) -> ProviderPlaneChart(Gamma')`
- `f*_A : AuditChart(Gamma) -> AuditChart(Gamma')`

and `Def` SHALL inherit pullback from these components.

### 10.1 Functoriality

Pullback SHALL satisfy:

- identity: `(id_Gamma)* X ~ X`
- composition: `(g o f)* X ~ g*(f*(X))`

for every chart object and every glued object.

### 10.2 Bridge pullback requirements

A conforming bridge pullback SHALL, when applicable:

- recompute effective authority as a meet under the narrower context,
- lower TTL by the appropriate minimum or tighter bound,
- intersect or otherwise narrow egress destinations,
- preserve or strengthen label restrictions,
- preserve single-use or human-review obligations when already required,
- and preserve burn.

### 10.3 Secret-core pullback requirements

A conforming secret-core pullback SHALL, when applicable:

- narrow version selection,
- preserve active-version predicates,
- preserve revocation or destruction,
- and never erase lifecycle history.

### 10.4 Materialization pullback requirements

A conforming materialization pullback SHALL, when applicable:

- lower or preserve TTL,
- narrow allowed methods or handle surfaces,
- preserve `non_exportable`,
- preserve `direct_model_access_prohibited`,
- preserve teardown obligations on expiry, burn, revocation, or consumer exit.

### 10.5 Provider-plane pullback requirements

A conforming provider-plane pullback SHALL, when applicable:

- refresh to equal or fresher authoritative facts,
- preserve provider attribution,
- fail closed on unverifiable fact substitution,
- and preserve more restrictive mode.

### 10.6 Audit pullback requirements

A conforming audit pullback SHALL, when applicable:

- preserve trace identity,
- project to a stream prefix or equivalent restriction,
- preserve hash linkage,
- and preserve source attribution.

## 11. Descent

### 11.1 Locality

For every declared cover of `Gamma`, restrictions of each chart object along
every covering map SHALL exist.

### 11.2 Contractible descent

For every cover `U` generated by chart projections or admissible refinements, the
restriction map:

- `res_U : Def(Gamma) -> Desc_U(Gamma)`

SHALL be an equivalence in `Gpd`.

Equivalently, every compatible descent datum over the cover SHALL glue to a
global admissible object uniquely up to isomorphism.

### 11.3 Finite-limit closure

Because this profile defines `Def` by finite-limit compatibility of the charts,
an implementation MAY prove descent for `Def` by proving:

1. descent for each chart, and
2. pullback-stability of the compatibility laws.

### 11.4 Refinement closure

Descent SHALL remain valid under admissible refinement of covers.

Checking only the coarsest five-chart cover is not sufficient if additional
locally admissible refinements change the meaning.

## 12. Optional pushforwards

Core conformance for this profile does not require any `Sigma_f` or `Pi_f`
surfaces.

A conforming implementation that does not expose them SHALL say so explicitly in
its conformance output.

### 12.1 Optional `Audit-Sigma` extension

An implementation MAY expose `Sigma_f` for audit aggregation.

If it does, `Sigma_f` SHALL be used only for evidence accumulation, checkpoint
roll-up, or equivalent audit transport, and SHALL satisfy the Premath kernel's
adjoint and Beck–Chevalley obligations.

`Sigma_f` SHALL NOT be used to widen authority or globalize plaintext.

### 12.2 Optional `Obligation-Pi` extension

An implementation MAY expose `Pi_f` for universal obligations, such as:

- audit-required,
- teardown-on-burn,
- no-plaintext-in-audit,
- freshness bounds,
- and non-exportability preservation.

If exposed, it SHALL satisfy the Premath kernel's adjoint and Beck–Chevalley
obligations.

## 14. Conformance tests

A conforming implementation SHOULD run at least the following tests.

1. Identity pullback on each chart and on `Def`.
2. Composition of pullback on each chart and on `Def`.
3. Glue success for a compatible five-chart cover.
4. Obstruction fixture for a denied or burned context that still emits durable
   evidence.
5. Glue failure for mismatched trace or secret seam.
6. Burn dominance under any chart entering burn.
7. TTL monotonicity under repeated refinement.
8. No-plaintext-global-section check.
9. Audit-before-allow check.
10. Provider-fact freshness rejection check.
11. Refinement-closure check on a refined cover.

## 15. Rejection criteria

An implementation is not conformant to this profile if it:

- treats raw provider payloads as the admissible provider-plane object without
  normalization,
- treats restore as an ordinary inverse of burn,
- widens authority under pullback,
- glues incompatible local sections,
- emits a global definable containing raw plaintext secret values,
- omits explicit resource-to-secret seam data,
- or silently advertises unavailable pushforward structure.

## Appendix A. Mapping to the current bundle

This profile is designed to sit on top of the current uploaded suite as follows.

- `BridgeChart` covers Bridge Spec v0.2, executable verifier, and adapter
  policy-input assembly.
- `SecretCoreChart` covers SECRET-0001.
- `MaterializationChart` covers SECRET-0002.
- `ProviderPlaneChart` covers SECRET-0003.
- `AuditChart` covers the durable audit surfaces in SECRET-0003.

The current suite already has most of the required objects. The main additions in
this profile are:

- an explicit shared context object,
- an explicit context-morphism object,
- explicit overlap objects,
- an explicit bridge-to-secret seam,
- and an explicit global glued object.

## Appendix B. Plaintext asymmetry

If an implementation wishes to model plaintext or local handles categorically, it
SHOULD do so in a separate leaf-local presheaf rather than in `Def`.

That presheaf MAY have sections over execution leaves and MAY intentionally have
no required global sections.
